📁 SafeSync-CLI (실시간 고신뢰성 멀티스레드 디렉토리 동기화 도구)

SafeSync-CLI는 리눅스 시스템 프로그래밍 기술을 활용하여 로컬 디렉토리의 변경 사항(생성, 수정, 삭제)을 실시간으로 감시하고, 네트워크 장애가 빈번한 환경에서도 데이터를 안전하게 보존하기 위해 비동기 멀티스레딩 소켓 전송 및 이어받기(Partial Sync) 프로토콜을 적용한 고신뢰성 원격 백업 솔루션입니다.

본 프로젝트는 주석을 제외한 순수 C 소스 코드 라인 수 500+ 줄(실측 639줄)을 만족하며, 모든 기능이 모듈화되어 구현되었습니다.

 1. 시스템 아키텍처 및 설계 메커니즘

SafeSync-CLI는 복잡한 동시성 연산과 실시간 데이터 흐름을 정교하게 제어하기 위해 생산자-소비자(Producer-Consumer) 디자인 패턴을 핵심 구조로 하여 개발하였습니다.

1.1  실시간 디렉토리 감시 엔진 

lstat, opendir, readdir 시스템 콜을 결합하여 지정된 감시 대상 폴더 내의 모든 엔트리를 재귀적으로 스캔하고 파일 메타데이터(수정 시간, 크기 등)를 영구 기록과 대조합니다.

주요 튜닝 포인트 (터미널 가독성 극대화):

감시 엔진은 시스템 자원 낭비를 방지하기 위해 2초마다 폴링(Polling)을 반복하도록 안전하게 설정되어 있습니다.

디버깅 시 터미널 화면이 무의미한 주기 로그로 도배되는 것을 막기 위해, "변경 사항이 없을 때 2초마다 찍히는 일반 폴링 로그"는 코드를 주석 처리하여 가독성을 극대화했습니다. 이제 변경 사건이 실제로 포착되었을 때만 핵심 로그가 인라인 형태로 출력됩니다.

1.2  Thread-Safe Task Queue 

워커 스레드 간의 자원 경쟁(Race Condition)을 원천 차단하기 위해 POSIX Mutex(pthread_mutex_t)와 조건 변수(pthread_cond_t)를 적용한 C 언어 대기열 큐를 설계하였습니다.

변경 이벤트가 감지되면 대기열에 집어넣고(push), 대기하던 스레드 디스패처가 이를 깨워(signal) 안전하게 업무를 인계받습니다.

1.3  전송 오버플로우 방지 스냅샷 전송 

가장 어려웠던 버그 해결 내역: 대용량 파일(예: dd 명령어로 작성 중인 파일)을 백업 디렉토리에 쓰는 도중에 감시 스레드가 먼저 반응하면, 파일이 여전히 디스크에 실시간으로 작성되는 동안 전송이 시작되는 문제가 발생했습니다. 이로 인해 소켓 전송 엔진이 최초에 인지했던 파일 크기보다 실제 보낸 바이트 수가 계속 커지며 진행률이 100%를 무한정 초과(예: 181% 완료)하고 세션이 중단되는 데이터 오염 현상이 발생했습니다.

해결 알고리즘 구축: 이를 해결하기 위해 스냅샷 제어 알고리즘을 개발했습니다. 전송 스레드가 기동되는 시점의 파일 크기를 상한선(EOF)으로 동적 고정하고, total_sent가 최초 크기에 도달하면 즉시 전송을 중단하도록 제어했습니다. 다 쓰이지 못한 파일의 나머지 뒷부분은 파일 쓰기가 완료된 후, 다음 2초 주기 스캔 시 Watcher가 크기 변경을 다시 감지하여 이어받기 프로토콜을 통해 잔여 데이터를 안전하게 서버에 이어 붙이도록 구현을 고도화했습니다.

1.4  끊김 없는 이어받기(Partial Sync) 엔진

클라이언트와 서버가 파일 전송 시작 시 고유 규격인 FileHeader 구조체(파일명 길이, 파일 전체 크기)를 송수신합니다.

서버 측에 이미 존재하거나 불완전하게 쓰인 파일이 있다면, 서버가 현재까지 저장한 바이트 오프셋(Offset)을 클라이언트에 피드백(Handshake)합니다.

클라이언트는 lseek 시스템 콜을 통해 파일 지시자를 해당 오프셋으로 신속하게 이동한 뒤, 끊겼던 바이트 이후부터 파일 데이터를 전송하여 네트워크 대역폭 낭비를 완전히 보장합니다.

1.5  고급 커널 시그널 예외 제어

Graceful Shutdown (SIGINT - Ctrl+C): 프로세스가 즉각 강제 종료될 경우 발생하는 소켓 버퍼 유실 및 리눅스 시스템 포트 물림 문제를 해결하기 위해 Graceful Shutdown을 구축했습니다. 버퍼를 안전하게 비우고 작동 중인 세션을 정리한 뒤, sync.log 파일에 정확한 시스템 종료 타임스탬프를 로깅하고 종료합니다.

실시간 비동기 모니터링 (SIGUSR1): 사용자가 외부 터미널에서 kill -SIGUSR1 <PID> 시그널을 주입하면, 공유 스레드 상태 데이터베이스(g_worker_statuses)에 스레드 세이프하게 접근하여 현재 전송 중인 모든 파일의 진행 상황을 가독성 높은 동적 프로그레스 바 차트 형태로 터미널에 즉각 인쇄합니다.

연결 단절 완벽 복구 (SIGPIPE 무시): 대용량 파일 전송 중 서버가 비정상적으로 꺼지면 클라이언트가 강제 종료(Crash)되는 리눅스 기본 예외 상황을 극복하기 위해 SIGPIPE 시그널을 커널단에서 강제 무시(SIG_IGN) 처리했습니다. 이를 통해 서버가 꺼지더라도 클라이언트는 안전하게 다운 타임을 버티며 폴더 감시를 계속하게 됩니다.

 4. 실전 데모 시연 매뉴얼 

시연용으로 가상머신(VM) 터미널 창 3개를 바둑판 형태로 고르게 배열하여 사용합니다.

+------------------------------------+------------------------------------+
|                                    |                                    |
|             [터미널 1]             |             [터미널 2]             |
|         SafeSync 백업 서버         |       SafeSync 감시 클라이언트     |
|             (./server)             |      (./safesync ~/sync_test 2)    |
|                                    |                                    |
+------------------------------------+------------------------------------+
|                                                                         |
|                                [터미널 3]                               |
|                       명령어 수동 제어 및 시그널 주입 창                |
|                                                                         |
+-------------------------------------------------------------------------+


0) 초기화 세팅 (터미널 3에서 한 번 실행)

pkill -9 server; pkill -9 safesync
rm -rf ~/sync_test/*
rm -rf ./server_sync_folder/*
mkdir -p ~/sync_test


1단계: 수신 백업 서버 기동 (터미널 1)

./server


출력 확인: [SafeSync 수신 서버] 클라이언트 접속을 무한 대기합니다... 포트 8080을 활성화하고 서버는 온전하게 리스닝 블로킹 상태로 진입합니다.

2단계: 클라이언트 감시자 가동 (터미널 2)

./safesync ~/sync_test 2


가독성 체크 포인트: [Watcher] 초기 탐색 완료... 문구가 뜨고 나면, 2초마다 작동하는 감시 로그가 주석 처리되었기 때문에 터미널에 아무 문구도 출력되지 않고 완전히 깨끗한 상태를 유지합니다. 오직 실제 파일 변경이 발생할 때만 즉각 출력 루프가 반응합니다.

3단계: 대용량 파일 복사 및 스냅샷 전송률 확인 (터미널 3 -> 터미널 2)

dd if=/dev/urandom of=~/sync_test/bigfile.bin bs=1M count=50


터미널 2 확인: dd 명령어로 파일이 써지기 시작하자마자 감시 스레드가 신속하게 동작하여 동적 스냅샷 바이트 크기를 캡처하고, bigfile.bin -> X% 완료 진행률 표시가 100% 한계치 내에서만 부드럽게 올라갑니다.

4단계: 실시간 동적 스레드 리포트 시그널 주입 (터미널 3 -> 터미널 2)

파일 전송이 점진적으로 올라가고 있을 때 다른 통제 창에서 SIGUSR1 시그널을 비동기로 주입합니다.

kill -SIGUSR1 $(pidof safesync)


터미널 2 결과: 비동기 스레드 모니터 데이터가 Mutex 임계 동기화 영역을 거쳐 실시간 그래프로 터미널에 print 됩니다.

==================================================
📊 [SafeSync 실시간 스레드 진행 상황 리포트]
==================================================
[Worker 스레드 #1]
  - 파일명: bigfile.bin
  - 진행률: [=========>           ] 45%
  - 전송량: 23592960 / 52428800 Bytes
==================================================


5단계: 서버 오프라인 복원력 및 이어받기(Partial Sync) 검증

파일이 전송 중인 틈을 타서 [터미널 1 (서버)]에서 Ctrl + C를 누릅니다.

복원력 검증: 서버가 완전히 다운되어도, [터미널 2 (클라이언트)]는 SIGPIPE가 무시 처리되었기 때문에 강제 세그멘테이션 에러로 죽지 않고 꿋꿋이 살아서 감시 루프를 지킵니다.

다시 [터미널 1] 서버를 실행해 줍니다: ./server

[터미널 3]에서 파일 데이터에 작은 변경을 주어 감시 엔진을 재자극합니다: echo "re-trigger" >> ~/sync_test/bigfile.bin

결과: 클라이언트 터미널에 [Partial Sync] 'bigfile.bin' 파일을 XXXXX 바이트부터 이어서 전송합니다. 메시지가 실시간으로 활성화되며 처음부터가 아닌 끊겼던 40~50% 영역부터 전송률이 이어서 완벽히 채워집니다.

 5. 세션 로그 파일 데이터 검증 (sync.log)

프로그램의 상태 안정성 기록 여부를 검증해보기 위해 터미널 3에서 동기화 작업 후 최종 생성된 세션 파일을 열어봅니다.

cat sync.log


실제 로깅 내역 예시:

[2026-06-19 21:35:45] SafeSync 시스템이 사용자에 의해 정상적으로 Graceful Shutdown 되었습니다.


타임스탬프와 함께 시스템 기동 사건과 우아한 정상 종료(Graceful Shutdown) 사건이 순차적으로 추적되어 기록됩니다.
